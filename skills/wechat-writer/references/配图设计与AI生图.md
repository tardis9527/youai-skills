# 配图设计与 AI 生图

> 阶段 8 使用。基于定稿内容，以资深视觉设计师视角设计配图方案，产出中文 AI 出图提示词，并可直接调用兔子 API（tu-zi.com）的 GPT-Image-2 生图接口把提示词变成真实图片。

## 一、设计师视角：配图设计方法论

公众号配图不是随便配一张"好看的图"，而是要为文章的核心情绪和记忆点服务。设计前先问自己：

- 这篇文章最想让读者记住的一个画面/意象是什么？
- 用途是什么：封面图（决定分享卡片观感、影响点击）？文中插图（辅助理解、调节阅读节奏）？
- 目标读者的审美偏好是什么（专业人群偏克制 / 年轻读者偏活泼 / 情感向偏温暖）？
- 风格从内容本身推导，**禁止套用"AI 生图默认风格"**（如千篇一律的紫蓝渐变、通用赛博朋克、廉价卡通感），风格关键词要让人一眼看出"这是为这篇文章定制的"，而不是随便一张网图。

常见风格方向（按内容调性选择，不要机械匹配）：

- **编辑摄影感**（真实光影、纪实构图）—— 适合故事叙事、深度评论类文章
- **扁平插画 / 信息图** —— 适合干货教程、方法论类文章
- **极简线条 / 留白** —— 适合认知深度、观点类文章
- **复古胶片 / 暖色调** —— 适合怀旧、个人化叙事

## 二、每张图要确定的信息

| 维度 | 说明 |
|---|---|
| 用途 | 封面图 / 文中插图 / 结尾图 |
| 画面主体 | 具体是什么（人物 / 物品 / 场景 / 抽象意象） |
| 场景与构图 | 环境、视角（俯视 / 平视 / 特写）、留白位置（封面图需为标题预留文字区） |
| 色调与光影 | 主色调、明暗基调 |
| 风格关键词 | 3–6 个，不堆砌互相矛盾的风格词 |
| 比例 | 见第四节尺寸表 |

## 三、中文出图提示词公式（GPT-Image-2 优化）

```
[画面类型] + [画面主体与场景] + [构图与视角] + [色调与光影] + [风格关键词] + [画面比例/留白要求] + [负面提示]
```

示例（封面图，故事叙事类文章）：

> 一张公众号文章封面图，画面主体是深夜写字楼里一扇亮着灯的窗，俯视城市街道的视角，远处霓虹微光，近处窗台放着一杯已经凉掉的咖啡。整体色调是深蓝与暖黄的对比，光影克制、有电影感。风格：纪实摄影质感、浅景深、轻微颗粒感。横版构图，画面上方三分之一留白供标题文字使用。避免：卡通感、过度饱和、水印、人物正脸特写、文字乱码。

负面提示统一建议包含：低分辨率、模糊、水印、文字乱码、多余肢体、与整体色调不符的杂色。

## 四、参考尺寸表（已按 gpt-image-2 约束验证，可直接使用）

| 用途 | 比例 | 普通 size | 高清 size |
|---|---|---|---|
| 方图（头图/小图/卡片图） | 1:1 | `1024x1024` | `1536x1536` |
| 横版封面（公众号首图） | ~16:9 | `1152x656` | `1728x992` |
| 竖版配图（文中竖图/海报） | 3:4 | `768x1024` | `1152x1536` |

如需其他比例，需满足 gpt-image-2 的硬性约束（用于自行计算）：

- 宽、高都必须是 16 的倍数
- 最长边不超过 3840px
- 宽高比不超过 3:1
- 总像素需在 655,360 ~ 8,294,400 之间

## 五、API Key 获取、验证与存储

获取地址：`https://api.tu-zi.com/token`（登录后创建/复制 API Key，格式以 `sk-` 开头）。

**检查顺序**（从上到下，找到即用，不用继续往下找）：

1. 环境变量 `TUZI_API_KEY`
2. 项目目录下 `.tuzi-skills/.env`
3. 用户目录下 `~/.tuzi-skills/.env`

**都没有找到时**：

- 直接用文本提示用户粘贴 Key（不要用选择题形式），例如："检测到还没有配置兔子 API Key，请粘贴你的 Key（在 https://api.tu-zi.com/token 获取，以 sk- 开头）："
- 拿到 Key 后，**不额外发一次测试请求**——直接用本次要生成的第一张图作为验证：
  - 请求成功（HTTP 200 且返回图片数据）→ Key 有效，进入保存流程
  - 返回 401/403 等鉴权错误 → 判定 Key 无效，提示用户确认后重新粘贴，本次不写入本地
- 验证成功后，询问保存位置（默认推荐"项目级"）：
  - 项目级：写入 `.tuzi-skills/.env`（当前项目目录下，仅本项目可用）
  - 用户级：写入 `~/.tuzi-skills/.env`（用户主目录下，所有项目共享）
  - 写入方式为追加而非覆盖：`mkdir -p <目标目录>/.tuzi-skills && echo "TUZI_API_KEY=<用户提供的Key>" >> <目标目录>/.tuzi-skills/.env`
- 若当前目录是 git 仓库且 `.gitignore` 中未包含 `.tuzi-skills/`，主动提醒用户添加，避免 Key 被误提交到仓库。
- 用户之后如果说"换一个 key"/"重新设置 key"/"key 不对"，重新走一遍"索要 → 验证 → 保存"流程，覆盖旧值。

**使用已保存的 Key 时**，从文件读入环境变量再以变量形式引用，避免把明文 Key 直接写在可见的命令里：

```bash
export $(grep TUZI_API_KEY .tuzi-skills/.env 2>/dev/null || grep TUZI_API_KEY "$HOME/.tuzi-skills/.env" | xargs)
```

## 六、接口调用格式（OpenAI 兼容）

**Endpoint**：`POST https://api.tu-zi.com/v1/images/generations`

**Headers**：

```
Content-Type: application/json
Authorization: Bearer $TUZI_API_KEY
```

**请求体**（`gpt-image-2`，纯文生图）：

```json
{
  "model": "gpt-image-2",
  "prompt": "<第三节生成的中文提示词>",
  "size": "1152x656"
}
```

> 注意：`gpt-image-2` 在兔子 API 网关上**不需要、也不要传 `quality` 字段**（`quality` 只用于该渠道的 gemini 系模型），清晰度完全由 `size` 的像素值决定，按第四节尺寸表选择"普通"或"高清"档。

**curl 示例**：

```bash
curl -s -X POST "https://api.tu-zi.com/v1/images/generations" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $TUZI_API_KEY" \
  -d '{
    "model": "gpt-image-2",
    "prompt": "……",
    "size": "1152x656"
  }' -o response.json
```

**响应处理**（二选一，需都处理）：

```json
{ "data": [ { "b64_json": "<base64 字符串>" } ] }
```

或

```json
{ "data": [ { "url": "https://.../xxx.png" } ] }
```

- 若有 `b64_json` → base64 解码后直接写入本地图片文件
- 若有 `url` → 下载该地址内容保存为本地图片文件

## 七、错误处理

| 情况 | 判断依据 | 处理方式 |
|---|---|---|
| Key 无效 | HTTP 401 / 403 | 提示用户 Key 无效或余额不足，重新索要，不写入本地 |
| 内容被拒 | 返回的 `revised_prompt` 字段包含 `PROHIBITED_CONTENT` | 告知用户内容被拒，建议调整提示词中的敏感描述后重试 |
| 未生成图片 | 返回的 `revised_prompt` 字段包含 `NO_IMAGE` | 提示提示词过于模糊，建议补充画面细节（主体/场景/构图）后重试 |
| 接口/网络错误 | 非 2xx 且非上述情况 | 展示错误信息，询问是否重试一次 |
| 无 Bash/网络调用能力 | 当前为纯对话环境 | 降级为只输出提示词，提示用户自行前往支持 GPT-Image-2 的工具生成 |

## 八、输出与保存

- 保存目录：`output/wechat-images/`（不存在则自动创建）
- 文件命名：`{日期YYYYMMDD}_{序号}_{图片用途简述}.png`（如 `20260706_01_封面图.png`）
- 每张图生成后，展示本地保存路径，并简述画面内容供用户核对是否需要重新生成
